## 摘要

本项目完成了一个名为 **Changmusic** 的音乐信息系统。项目以酷我音乐公开页面及接口返回的数据为基础，使用 Python 编写爬虫，分阶段采集歌曲、歌词和歌手信息；随后使用 Django 构建 Web 信息展示系统，实现歌曲与歌手浏览、详情查看、关键词搜索、分页、评论和后台管理等功能；最后使用 Pandas、正则表达式和 Matplotlib 对歌词中的作词、作曲署名进行提取与分析，从而得出三个有意义的结论。

项目最终获得 **2214 首歌曲**和 **181 位歌手**的结构化数据。通过正则表达式，从歌词文本中识别作词人与作曲人，作词信息识别率为 **88.80%**，作曲信息识别率为 **89.48%**，词曲均成功识别的比例为 **88.62%**。在此基础上，本文围绕“创作者资源集中程度”“歌手自我创作程度”“歌手与核心创作者的稳定合作关系”三个问题展开分析。

分析结果表明：第一，歌曲创作呈现“头部创作者较为集中、长尾创作者数量庞大”的结构；第二，多数歌手主要承担演唱角色，但少数唱作型歌手深度参与词曲创作，且总体上歌手参与作曲的程度高于参与作词；第三，部分歌手与固定作词人、作曲人或组合内部核心成员形成了稳定合作关系，但高度依赖单一创作者的歌手仍属于少数。

**关键词：** Python；网络爬虫；Django；Pandas；Matplotlib；正则表达式；音乐数据分析

## 4. 数据采集与处理

### 4.1 歌曲数据采集

歌曲采集脚本以歌手姓名作为搜索关键词，请求酷我音乐歌曲搜索接口，并从返回的 JSON 中读取歌曲信息。主要字段包括歌曲名、歌手名、歌手 ID、歌曲 ID、MusicRID、专辑、时长、封面和来源链接。

为了避免同一首歌因多个关键词重复出现，程序使用集合保存已经出现过的 `song_id`：

```python
seen_song_ids = set()
```

当新歌曲的 ID 已存在时，将其记录为重复数据并跳过；否则加入结果列表。该方法利用集合查找速度快的特点，能够较高效地完成去重。

对于缺失字段，程序进行了基础处理：

- 缺少歌曲 ID 时直接跳过；
- 缺少或无效歌手 ID 时跳过；
- 专辑为空时填入“无专辑归属”；
- 封面为空时使用默认图片。

### 4.2 歌词数据采集

歌词通过歌曲 ID 请求歌词接口，并读取返回结果中的逐行歌词，再使用换行符拼接成完整文本。

为提高长时间运行时的稳定性，歌词采集采用了以下措施：

1. 使用 `try/except` 捕获网络异常；
2. 对空歌词进行检查并记录删除原因；
3. 已存在歌词时直接跳过，避免重复请求；
4. 每处理 20 首歌曲进行一次阶段性保存；
5. 使用 `time.sleep()` 控制请求频率；
6. 检测到已有阶段文件时从已有数据继续处理，实现简单断点续爬。

最终只保留具有有效歌词的歌曲，为后续创作者署名提取提供数据基础。

### 4.3 歌手数据采集

歌手信息来自酷我歌手详情页。该页面返回 HTML，其中歌手信息位于包含 `singerInfo` 的 JavaScript 脚本中。

处理流程如下：

1. 使用 Requests 获取歌手详情页 HTML；
2. 使用 BeautifulSoup 查找所有 `<script>` 标签；
3. 定位包含 `singerInfo` 的脚本文本；
4. 使用正则表达式提取字符串字段和数字字段；
5. 使用 `json.loads()` 和 `html.unescape()` 还原转义字符；
6. 缺少生日或国家信息时填入“暂无信息”。

这种方式综合使用了 HTML 解析和正则表达式，解决了数据并非直接出现在普通 HTML 标签中的问题。

### 4.4 数据质量控制

项目在数据采集和分析阶段主要进行了以下质量控制：

- 按歌曲 ID 去重；
- 检查歌曲与歌手关键字段；
- 过滤空歌词；
- 设置请求超时并调用 `raise_for_status()`；
- 对采集结果阶段性保存；
- 对歌手与歌曲关联进行检查；
- 对作词、作曲识别率进行统计；
- 对多位创作者进行拆分，并删除空关系；
- 统计时使用 `nunique()`，避免同一首歌曲被重复计算。

## 5. Django 音乐信息系统设计与实现

### 5.1 数据模型

系统围绕三个核心实体组织数据：

- **Artist**：保存歌手基本信息；
- **Song**：保存歌曲、专辑、歌词、封面和来源链接等信息；
- **Comment**：保存歌曲评论、昵称和创建时间。

歌曲与歌手之间建立关联，评论与歌曲之间建立一对多关系。对于未成功关联到歌手对象的歌曲，页面仍保留原始歌手名，避免详情页报错。

### 5.2 路由与视图

系统主要路由包括：

```text
/                              首页
/songs/                        歌曲列表
/songs/<id>/                   歌曲详情
/artists/                      歌手列表
/artists/<id>/                 歌手详情
/search/                       综合搜索
/comments/<id>/delete/         删除评论
/admin/                        Django 后台
```

视图函数负责接收请求、查询数据库、分页、处理评论表单，并将结果传递给模板。

### 5.3 搜索实现

搜索功能使用 Django ORM 的 `Q` 对象组合多个模糊匹配条件。例如歌曲搜索逻辑为：

```python
Q(song_name__icontains=query)
| Q(artist_name__icontains=query)
| Q(lyrics__icontains=query)
```

其中 `icontains` 表示不区分大小写的包含匹配，多个条件之间使用逻辑或连接。

系统使用 `perf_counter()` 记录后端搜索开始和结束时间，并强制读取当前页数据，使计时包含数据库实际查询过程。

### 5.4 分页实现

系统使用 Django 的 `Paginator`：

- 歌曲列表每页 20 首；
- 歌手列表每页 24 位；
- 歌手详情中的相关歌曲每页 12 首；
- 综合搜索结果每页 20 条。

页面提供首页、上一页、下一页、末页和指定页码跳转功能。在搜索结果中，翻页链接会保留关键词和搜索类型。

### 5.5 模板与界面设计

页面采用 Django 模板继承机制，以 `base.html` 作为基础模板，统一导航栏、页面容器和页脚。各功能页面通过模板块填充具体内容。

前端使用 HTML 和 CSS 实现卡片、表格、搜索框、分页、图片展示和响应式布局。系统界面以信息清晰和功能易用为主要目标，未引入额外前端框架，便于理解模板与后端数据之间的对应关系。

## 6. 数据分析方法

### 6.1 作词、作曲信息提取

原始歌曲数据没有独立的作词人和作曲人字段，但歌词开头经常出现如下署名：

```text
作词：方文山
作曲：周杰伦
Lyrics by: Michael Jackson
Composed by: Michael Jackson
```

因此，项目使用正则表达式识别以下格式：

- `作词`、`作词人`、`填词`、`词`；
- `作曲`、`作曲人`、`谱曲`、`曲`；
- `词曲`、`作词作曲`、`词/曲`；
- `Lyrics by`；
- `Composed by`。

创作者信息通常位于歌词开头，因此程序只检查前若干行，并清除形如 `[00:12.35]` 的时间标签。

识别结果如下：

| 指标 | 结果 |
|---|---:|
| 作词识别率 | 88.80% |
| 作曲识别率 | 89.48% |
| 词曲均识别率 | 88.62% |
| 至少一项识别失败 | 252 首 |

识别率接近 90%，能够支持后续统计分析。剩余未识别数据主要来自歌词中没有标准署名或使用了未覆盖的特殊格式。

### 6.2 多人创作者拆分

一首歌可能包含多位作词人或作曲人，例如：

```text
古小力/黄淩嘉
刘涛、温莨、董临
```

程序使用正则表达式按照斜杠、顿号、逗号和分号等分隔符拆分姓名，并使用：

```python
list(dict.fromkeys(names))
```

在保持原顺序的同时去除同一首歌中的重复姓名。

随后使用 Pandas 的 `explode()` 将姓名列表展开为多行，从而构造“一首歌曲—一位创作者”的长表。最终得到 2585 条作词关系和 2623 条作曲关系。

### 6.3 主要 Pandas 操作

分析过程中使用的主要 Pandas 方法包括：

| 方法 | 作用 |
|---|---|
| `apply()` | 对一列元素或每一行执行函数 |
| `explode()` | 将列表列展开为多行 |
| `groupby()` | 按歌手或创作者分组 |
| `agg()` | 对分组结果执行多项汇总统计 |
| `nunique()` | 统计不同歌曲 ID 的数量 |
| `reset_index()` | 将分组索引恢复为普通列 |
| `sort_values()` | 按统计值排序 |
| `isin()` | 判断值是否属于指定集合 |
| `merge()` | 按共同字段合并两张表 |
| `value_counts()` | 统计分类值出现次数 |

## 7. 数据分析结果

### 7.1 结论一：创作资源呈现“头部集中、长尾庞大”的结构

首先按照创作者分组，统计每位作词人和作曲人参与的不同歌曲数量。

作词人中，林夕参与 70 首歌曲，李宗盛参与 49 首，姚若龙参与 43 首，方文山参与 34 首。作曲人中，周杰伦参与 43 首，李宗盛参与 41 首，林俊杰参与 38 首。

> **此处插入图 8：作词歌曲数 Top 10 横向柱状图**  
> **此处插入图 9：作曲歌曲数 Top 10 横向柱状图**

在成功识别的数据中，共统计到 815 位作词人和 990 位作曲人。前 10 位作词人仅占全部作词人的约 1.23%，却覆盖了 16.99% 的已识别作词歌曲；前 10 位作曲人仅占全部作曲人的约 1.01%，却覆盖了 14.54% 的已识别作曲歌曲。

这说明创作机会并非平均分布，少数头部创作者参与了远高于平均水平的歌曲数量。其中，作词头部的集中程度略高于作曲头部。

不过，Top 10 以外仍存在 805 位作词人和 980 位作曲人，其余创作者合计覆盖超过 80% 的歌曲，因此不能将该结构描述为少数人完全垄断。更准确的结论是：**创作者分布具有明显头部集中现象，同时存在规模庞大的长尾群体。**

### 7.2 结论二：歌手创作角色明显分化，参与作曲比参与作词更普遍

本部分将歌手姓名与该歌曲的作词人、作曲人列表进行比较：

- 歌手姓名出现在作词人列表中，记为自作词；
- 歌手姓名出现在作曲人列表中，记为自作曲。

为降低小样本影响，只保留作词和作曲有效歌曲数均不少于 5 首的歌手，共 126 位。

总体结果如下：

| 指标 | 结果 |
|---|---:|
| 总体自作词率 | 18.25% |
| 总体自作曲率 | 25.72% |
| 自作词率中位数 | 0.00% |
| 自作曲率中位数 | 0.00% |
| 两项均为 0 | 64 人 |
| 自作曲率明显更高 | 28 人 |
| 两项非零且接近 | 23 人 |
| 自作词率明显更高 | 11 人 |

按照 50% 的操作性标准进行分类：

| 类型 | 人数 | 比例 |
|---|---:|---:|
| 演唱主导型 | 96 | 76.19% |
| 综合唱作型 | 18 | 14.29% |
| 作曲主导型 | 12 | 9.52% |
| 作词主导型 | 0 | 0% |

> **此处插入图 10：歌手自作词率与自作曲率气泡散点图**

散点图中横轴为自作词率，纵轴为自作曲率，点大小表示歌手歌曲数量。大量点位于对角线上方，说明不少歌手的自作曲率高于自作词率。

具体来看，周杰伦在本样本中的自作曲率为 100%，自作词率为 16%；林俊杰的自作曲率约为 95.45%，自作词率为 0；王力宏的自作词率约为 59.09%，自作曲率约为 81.82%，属于综合唱作型。

综合来看，多数歌手在样本中主要承担演唱角色，少数唱作型歌手深度参与创作。同时，歌手参与作曲的程度整体高于参与作词，说明歌词创作可能更依赖专业作词人，而部分歌手会直接参与歌曲旋律创作。

### 7.3 结论三：稳定创作搭档存在，但高度集中合作只出现在少数歌手中

本部分构造“歌手—作词人”和“歌手—作曲人”关系表，并统计每组搭档共同参与的不同歌曲数量。为避免与上一部分重复，先排除歌手姓名与创作者姓名完全相同的记录。

随后，对每位歌手选取合作歌曲数最多的一位作词人和一位作曲人作为核心搭档，并计算：

```text 
合作占比 = 与核心搭档合作的歌曲数 ÷ 对应创作者信息有效的歌曲数
```

分析结果如下：

| 指标 | 作词搭档 | 作曲搭档 |
|---|---:|---:|
| 存在有效核心搭档的歌手 | 125 人 | 123 人 |
| 合作占比不低于 50% | 16.80% | 13.82% |
| 合作占比不低于 75% | 8.00% | 7.32% |

> **此处插入图 11：核心作词搭档与核心作曲搭档合作集中程度对比图**

典型作词合作关系包括：

- 五月天—阿信：19/20，合作占比 95%；
- 张宇—十一郎：15/17，合作占比约 88.24%；
- 周杰伦—方文山：15/25，合作占比 60%；
- 周传雄—陈信荣：13/21，合作占比约 61.90%。

典型作曲合作关系包括：

- BEYOND—黄家驹：16/16，合作占比 100%；
- 刘珂矣—百慕三石：15/16，合作占比约 93.75%；
- China Blue—伍佰：15/17，合作占比约 88.24%；
- 五月天—阿信：13/20，合作占比 65%。

这些关系包含两种模式：一类是歌手与固定外部创作者长期合作，例如周杰伦与方文山；另一类是乐队或组合内部由核心成员承担创作，例如五月天与阿信、BEYOND 与黄家驹。

总体而言，稳定创作搭档确实存在，但核心搭档负责一半以上作品的歌手比例不足 20%。因此，大多数歌手仍与多位创作者合作，只有少数歌手或组合形成了高度稳定的核心创作结构。固定作词搭档的集中程度略高于固定作曲搭档，但两者差距不大。

## 10. 实验体会与改进建议

### 10.1 实验体会

本项目最明显的体会是：一个完整的数据项目不仅是“把数据爬下来”，还包括数据质量控制、结构化存储、网页展示、用户交互和结果解释。

在爬虫阶段，最初容易只关注请求是否成功，但实际运行后会遇到重复歌曲、字段为空、接口格式变化、请求失败和程序中断等问题。通过集合去重、异常处理和阶段性保存，程序才具备较好的稳定性。

在 Django 开发阶段，对 MVT 结构有了更直观的理解：模型负责数据，视图负责业务逻辑，模板负责页面展示，URL 负责把请求分发到对应视图。搜索、分页和评论功能使数据库查询、请求参数和模板渲染真正连接起来。

在数据分析阶段，进一步认识到 Pandas 操作应从实际问题出发。`explode()`、`groupby()`、`agg()`、`merge()` 等函数不需要孤立记忆，而可以对应“展开多人名单”“按人分组”“汇总指标”“拼接两张表”等具体数据操作。

此外，分析结论不能只看图或只看单个比例。例如稳定搭档既要看合作占比，也要看实际合作歌曲数；歌手自创率还需要考虑有效样本数量和姓名匹配规则。数据分析不仅是计算，更重要的是定义指标并解释其适用范围。

### 10.2 当前不足

项目仍存在以下不足：

1. 数据来自特定歌手关键词和当前爬取范围，样本不代表平台全部音乐；
2. 作词、作曲提取依赖规则，仍有约 10% 的歌曲未能完整识别；
3. 创作者姓名尚未建立完整别名表，例如中文名、英文名、艺名和本名可能被视为不同人员；
4. 乐队与成员关系未被单独建模，可能影响自创率和外部合作关系判断；
5. 网站暂未实现用户账号、收藏、评分和个性化推荐；
6. 搜索采用数据库模糊匹配，数据规模更大时需要考虑全文索引；
7. 数据未实现定期自动更新。

## 11. 总结

本项目完成了从音乐数据采集、清洗、存储到 Web 展示和数据分析的完整闭环。系统最终收录 2214 首歌曲和 181 位歌手，实现了歌曲、歌手、搜索、分页、评论和后台管理等功能。

在数据分析部分，通过正则表达式从歌词中提取作词人与作曲人，并使用 Pandas 构造歌曲—创作者、歌手—创作者关系表，最终得到三个主要结论：

1. 创作者分布呈现头部集中与长尾并存的结构；
2. 歌手在音乐生产中的角色存在明显分化，且参与作曲比参与作词更普遍；
3. 少数歌手与核心创作者形成高度稳定的合作关系，但多数歌手仍具有较多样的创作来源。

通过本实验，综合运用了 Python 网络请求、HTML 与 JSON 解析、正则表达式、异常处理、Django Web 开发、数据库操作、Pandas 数据处理和 Matplotlib 可视化等知识。
